# 运维概览

OpenAI4S 是一套供**单一用户在本地或可信主机上使用的科学工作台**，不是面向公网的多租户应用服务器。支持的运行形态是：一名可信操作员、一个专用操作系统账户、一个数据目录，以及一个仅绑定回环地址的守护进程。远程用户通过 SSH 隧道或可信 VPN 访问这个回环监听器。

公开文档站点是另一个独立系统。`openai4s.org/docs/` 是静态 VitePress 构建产物，其中不包含工作台进程、模型密钥、SQLite 数据库或研究 Artifact，可以公开提供。不要在公开文档域名下反向代理工作台。

## 支持的运行形态

| 关注点 | 支持的基线 | 不应假设具备 |
|---|---|---|
| 使用者 | 一名可信操作员及其本地研究会话 | 租户隔离、逐用户授权、配额或管理员角色 |
| 网络 | `127.0.0.1:8760`；远程访问使用 SSH 隧道或可信 VPN | 可以安全地直接暴露到互联网 |
| 进程 | 一个前台守护进程，通常由操作系统监管 | 主动/主动副本，或多个守护进程并发共享同一数据目录 |
| 存储 | 一个本地 `OPENAI4S_DATA_DIR`，包含 SQLite 与文件状态 | 网络文件系统协调，或只备份数据库即可完整恢复 |
| 执行 | 守护进程账户下按需启动的 Python/R 子进程 | 面向恶意代码的多租户沙箱 |
| 可用性 | 由操作员负责重启与恢复 | 零停机升级或自动灾难恢复 |

守护进程通过 pidfile 保证单实例。所有服务器线程共享一个 `Store` 连接，而科学计算在子进程中运行。每套安装的数据目录应仅供一个守护进程账户私有使用，绝不能让两个正在运行的守护进程指向同一目录。

## 首次启动检查清单

1. 创建专用的禁止登录账户，或至少是权限尽可能小的操作系统用户。
2. 创建仅由该用户拥有的本地数据目录，并以 `umask 077` 启动服务。
3. 保持 `OPENAI4S_HOST=127.0.0.1`。如果沙箱不可用时必须终止执行，请使用 `OPENAI4S_KERNEL_SANDBOX=enforce`。
4. 以服务用户身份安装并测试发行版本。守护进程可能在后台为解释器安装缺失的科学计算包；确认服务就绪前，应等待环境状态稳定。
5. 配置模型凭据时不要把它们放入代码仓库。SQLite 数据库、`.env`、日志、会话导出和 Artifact 都应视为敏感数据。
6. 接受正式工作前，运行离线测试门禁，并通过真实浏览器对回环地址做冒烟测试。
7. 首次升级前，先制作一份服务已停止状态下的完整数据目录备份，并实际测试恢复流程。

## 日常检查

请从与服务相同的环境中运行 CLI，并使用相同的 `OPENAI4S_DATA_DIR`：

```bash
openai4s status
curl --fail --silent http://127.0.0.1:8760/health
```

同时检查：

- 数据目录的磁盘空间与 inode 使用量；
- 目录所有权，以及组用户/其他用户均无访问权限；
- 环境状态中是否存在后台科学包安装失败；
- Python 和 R 实际启动后的工作台安全状态投影，因为 worker 自检运行前，沙箱状态为 `not_started`；
- 待处理审批与非预期的长期权限规则；
- 未完成的远程作业；它们并不是持久化调度队列。

## 事件响应优先级

如果怀疑发生了暴露、凭据泄漏、状态损坏或非预期代码执行：

1. 停止守护进程和远程计算活动。
2. 保留数据目录与服务日志的只读副本。
3. 在 OpenAI4S 外部撤销模型、搜索、连接器、云端及远程计算凭据。
4. 检查权限、SSH 访问、近期 Action/Cell 与 Artifact 版本。
5. 仅在备份与已知代码发行版本匹配，并已在隔离的回环实例中验证后，才进行恢复。

恢复过程中不要删除原始数据目录。OpenAI4S 在数据库打开时执行前向迁移，因此回滚必须同时具备兼容的代码版本和升级前数据快照。
